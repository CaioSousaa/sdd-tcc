# Especificação: RF11 – Filtragem de Tarefas

## História de Usuário

**Como** um usuário autenticado do sistema,
**Quero** poder filtrar minha lista de tarefas por prioridade e/ou tags,
**Para que** eu possa organizar minha visualização e focar rapidamente nas tarefas mais relevantes ao meu contexto no momento.

## Decisões e Comportamento Esperado

O endpoint utilizado para esta funcionalidade será o `GET /tasks`, recebendo os parâmetros de filtro via *query string*.

### Critérios de Aceite e Cenários

1. **Busca sem filtros (Sucesso)**
   - **Dado** que o usuário está autenticado e possui tarefas cadastradas;
   - **Quando** realizar uma requisição válida sem informar parâmetros de filtro;
   - **Então** o sistema deve retornar status `200 OK`;
   - **E** deve retornar a lista completa de tarefas pertencentes ao usuário.

2. **Busca com filtros válidos (Sucesso)**
   - **Dado** que o usuário está autenticado;
   - **Quando** realizar uma requisição válida informando filtros de `priority` e/ou `tags`;
   - **Então** o sistema deve retornar status `200 OK`;
   - **E** deve retornar apenas as tarefas do usuário autenticado que correspondam exatamente aos critérios informados.

3. **Busca não retorna resultados (Sucesso)**
   - **Dado** que o usuário está autenticado;
   - **Quando** realizar uma requisição válida informando filtros de `priority` e/ou `tags`;
   - **Mas** nenhuma tarefa atende aos critérios;
   - **Então** o sistema deve retornar status `200 OK` com uma lista vazia `[]`.

4. **Acesso não autenticado (Erro)**
   - **Dado** que a requisição não possui um token de autenticação válido ou está ausente;
   - **Quando** tentar acessar a rota `GET /tasks`;
   - **Então** o sistema deve retornar status `401 Unauthorized`;
   - **E** deve retornar a mensagem padrão do sistema para falhas de autenticação.

5. **Filtro de prioridade inválido (Erro)**
   - **Dado** que o usuário está autenticado;
   - **Quando** enviar um valor para `priority` diferente de `low`, `medium` ou `high`;
   - **Então** o sistema deve retornar status `400 Bad Request`;
   - **E** deve retornar a mensagem de erro `"Prioridade inválida"`.

6. **Filtro de tag inexistente (Erro)**
   - **Dado** que o usuário está autenticado;
   - **Quando** informar o ID de uma tag no filtro que não existe no banco de dados;
   - **Então** o sistema deve retornar status `400 Bad Request`;
   - **E** deve retornar a mensagem de erro `"Tag não encontrada"`.

### Regras de Negócio

1. **Permissão**: Apenas usuários devidamente autenticados devem poder filtrar as tarefas.
2. **Escopo de Dados**: O sistema deve sempre retornar unicamente tarefas pertencentes ao usuário autenticado responsável pela requisição. Jamais tarefas de terceiros devem ser vazadas, independentemente dos filtros informados.
3. **Flexibilidade de Filtros**: Os parâmetros de filtro `priority` e `tags` são totalmente opcionais e podem ser combinados na mesma requisição.
4. **Intersecção de Filtros (AND)**: Quando ambos os filtros (`priority` e `tags`) forem enviados em conjunto na requisição, o sistema deve retornar apenas as tarefas que satisfaçam as duas condições simultaneamente.
5. **Validação de Propriedade da Tag**: O usuário tem a restrição de apenas poder filtrar por tags que estejam vinculadas à sua própria conta (tags de sua propriedade).
6. **Ausência de Resultados**: Se a combinação dos filtros ou um filtro isolado não tiver match com as tarefas cadastradas, o sistema deve processar normalmente retornando status HTTP `200 OK` contendo um array vazio.
