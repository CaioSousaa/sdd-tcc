# RF13-RF14-RF15 — Notificações, Marcação como Lida, Menu do Usuário

## Decisões

### RF13 — Criação Automática de Notificação
- A criação da notificação é interna: não há endpoint exposto para criá-la diretamente.
- O disparo ocorre quando o job agendado pelo RF12 executa no horário definido em `alert`.
- Antes de criar a notificação, o sistema verifica se a tarefa ainda existe; se não existir, o job é cancelado silenciosamente sem criar notificação.
- Antes de criar a notificação, o sistema verifica se o dono da tarefa ainda existe; se não existir, o job é cancelado silenciosamente sem criar notificação.
- O campo `message` é preenchido automaticamente com: `"Lembrete: a tarefa {título da tarefa} está aguardando sua atenção"`.
- O campo `read` é sempre `false` na criação.
- O campo `owner` é herdado da tarefa correspondente ao job.
- Cada disparo gera exatamente uma notificação.

### RF14 — Marcar Notificação como Lida
- Rota: `PATCH /notifications/:id/read`.
- Requer autenticação via JWT.
- Notificações de outros usuários retornam `404` (mesmo tratamento de inexistente) para não expor dados alheios.
- Notificação já marcada como lida retorna `400`; a operação não é reversível.

### RF15 — Listagem de Notificações, Atualização de Dados do Usuário e Logout
- **GET /notifications**: lista todas as notificações do usuário autenticado, ordenadas por `createdAt` decrescente. Lista vazia retorna `200` com `[]`.
- **PATCH /users/me**: permite atualizar `name` e/ou `password`. O campo `email` não pode ser atualizado por esta rota. Ao menos um campo deve ser enviado. Nova senha é armazenada com hash. `updatedAt` não faz parte do schema e não é retornado.
- **POST /auth/logout**: invalida o JWT do usuário. Após o logout, requisições com o token invalidado são rejeitadas com `401`. A invalidação é feita via blacklist em memória (conjunto de tokens invalidados verificado no middleware de autenticação).

## Histórias de Usuário

### RF13
**Como** sistema,  
**quero** criar uma notificação automaticamente quando o job de alerta de uma tarefa disparar,  
**para que** o usuário seja informado sobre a tarefa no horário definido, sem nenhuma ação manual.

### RF14
**Como** usuário autenticado,  
**quero** marcar uma notificação como lida,  
**para que** eu possa controlar quais alertas já foram visualizados.

### RF15
**Como** usuário autenticado,  
**quero** acessar um menu com minhas notificações, alterar meus dados e encerrar minha sessão,  
**para que** eu possa gerenciar minha conta e acompanhar alertas sem sair do sistema.

## Critérios de Aceite

### RF13 — Criação Automática de Notificação

1. **Job disparado com tarefa válida**: quando o horário de `alert` é atingido, o sistema cria automaticamente uma notificação com `read: false`, `owner` igual ao dono da tarefa, `task` igual à tarefa correspondente e `message` no formato `"Lembrete: a tarefa {título} está aguardando sua atenção"`.

2. **Job disparado para tarefa inexistente**: se a tarefa foi excluída antes do disparo, o sistema cancela o job sem criar notificação.

3. **Job disparado para dono inexistente**: se o usuário dono da tarefa foi excluído antes do disparo, o sistema cancela o job sem criar notificação.

4. **Uma notificação por disparo**: cada execução do job cria exatamente uma notificação; nenhum disparo gera zero ou mais de uma notificação.

5. **Sem endpoint exposto**: não existe rota pública para criar notificações diretamente.

### RF14 — Marcar Notificação como Lida

1. **Requisição válida**: `PATCH /notifications/:id/read` com token válido e notificação pertencente ao usuário retorna `200` com `{ "message": "Notificação marcada como lida" }` e atualiza `read` para `true`.

2. **Não autenticado**: requisição sem token ou com token inválido retorna `401` com a mensagem padrão do sistema.

3. **Notificação inexistente**: notificação com `:id` não encontrado no banco retorna `404` com `{ "message": "Notificação não encontrada" }`.

4. **Notificação de outro usuário**: notificação pertencente a outro usuário retorna `404` com `{ "message": "Notificação não encontrada" }`.

5. **Notificação já lida**: tentativa de marcar uma notificação que já possui `read: true` retorna `400` com `{ "message": "Notificação já marcada como lida" }`.

### RF15 — Listagem de Notificações

1. **Requisição válida**: `GET /notifications` com token válido retorna `200` com a lista de notificações do usuário autenticado, ordenadas por `createdAt` decrescente, incluindo o campo `read` em cada item.

2. **Sem notificações**: usuário sem notificações retorna `200` com `[]`.

3. **Não autenticado**: retorna `401` com a mensagem padrão do sistema.

4. **Isolamento**: a lista contém apenas notificações do usuário autenticado.

### RF15 — Atualização de Dados do Usuário

1. **Requisição válida**: `PATCH /users/me` com ao menos `name` ou `password` retorna `200` com `{ "message": "Dados atualizados com sucesso" }`.

2. **Não autenticado**: retorna `401` com a mensagem padrão do sistema.

3. **Nenhum campo enviado**: corpo vazio ou sem `name` nem `password` retorna `400` com `{ "message": "Nenhum campo para atualizar foi informado" }`.

4. **Senha fraca**: `password` com menos de 8 caracteres retorna `400` com `{ "message": "A senha deve ter no mínimo 8 caracteres" }`.

5. **Hash da senha**: quando `password` é enviado, ele é armazenado com hash antes de ser persistido; a senha em texto plano nunca é salva.

6. **Campo email bloqueado**: envio de `email` no corpo não causa atualização do e-mail do usuário.

### RF15 — Logout

1. **Requisição válida**: `POST /auth/logout` com token válido retorna `200` com `{ "message": "Logout realizado com sucesso" }` e invalida o token.

2. **Não autenticado**: retorna `401` com a mensagem padrão do sistema.

3. **Token invalidado**: após o logout, qualquer requisição com o mesmo token é rejeitada com `401`.
