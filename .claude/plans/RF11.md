# Plano de Implementação: RF11 – Filtragem de Tarefas

Com base na especificação gerada e no contexto do projeto, o plano abaixo define a estratégia técnica para implementar a filtragem de tarefas no backend, respeitando a arquitetura limpa (Clean Architecture).

## 1. Stack Tecnológica
- **Linguagem**: TypeScript 6
- **Framework**: Express 5
- **Banco de Dados**: MongoDB via Mongoose
- **Comunicação**: Frontend acessará a API REST utilizando instâncias do Axios.

## 2. Modelo de Dados (Mongoose)
O esquema do `Task` já deve possuir todos os campos necessários conforme os RFs anteriores:
- `owner`: `ObjectId` referenciando `User`.
- `priority`: `String` (enum restrito a `"low"`, `"medium"`, `"high"`).
- `tags`: Array de `ObjectId` referenciando `Tag`.

*Não será necessário criar ou alterar o schema atual do banco de dados, usaremos apenas recursos de query do Mongoose.*

## 3. Estrutura do Endpoint

**Endpoint:** `GET /tasks`

**Parâmetros Esperados (Query Params):**
- `priority` (opcional): `string`
- `tags` (opcional): `string` (uma única tag) ou `array de strings` (múltiplas tags) contendo ObjectIds.

**Fluxo de Execução:**

1. **Controller (`TaskController.ts`)**:
   - Recebe a requisição após o middleware de autenticação (`req.userId` já preenchido).
   - Extrai de `req.query` as propriedades `priority` e `tags`.
   - Garante que a propriedade `tags`, se vier, seja parseada num array de strings uniformemente.
   - Invoca a chamada do UseCase / Service (ex: `ListUserTasksService.execute`).
   - Retorna as instâncias (JSON) recebidas com status `200`.

2. **Service (`ListUserTasksService.ts`)**:
   - **Validação de Prioridade:** Se a variável de prioridade estiver presente e não for `"low"`, `"medium"` ou `"high"`, lançar uma exceção de aplicação (`AppError` ou similar) contendo a mensagem `"Prioridade inválida"`.
   - **Validação de Tags:** Se existirem `tags`, será necessário verificar se de fato existem e pertencem ao usuário. Isso pode ser feito injetando um repositório de tags (`ITagRepository.findByIds`) para checar se a quantidade de tags retornadas com o `owner === userId` bate com a quantidade solicitada. Caso falhe, dispara o erro `"Tag não encontrada"`.
   - Invoca o `TaskRepository.findUserTasks(userId, filters)`.

3. **Repository (`TaskRepository.ts` e Port `ITaskRepository.ts`)**:
   - O repositório receberá o ID do proprietário obrigatoriamente e os filtros (`{ priority?, tags? }`).
   - Mongoose Query base: `const query = { owner: userId };`
   - Se for enviada `priority`, adicionar: `query.priority = priority;`
   - Se for enviadas `tags`, utilizar o operador `$in` (para "qualquer tag fornecida") ou `$all` (para "exigir todas as tags fornecidas"). Pela flexibilidade de filtro, utilizaremos o operador `$in`: `query.tags = { $in: tagsArray };`.

## 4. Tratamento de Erros
A aplicação já utiliza um tratamento padrão, conforme o fluxo do Express:
- **`401 Unauthorized`**: Tratado no começo pela chain de Middlewares (`shared/http/authenticate.ts`). Apenas o erro padrão vai subir se não houver um token válido.
- **`400 Bad Request`**:
  - Quando a prioridade falhar: Disparo de erro com mensagem exata (`"Prioridade inválida"`) a ser capturada pelo Handler global de erro, que o mapeia para o HTTP code correto.
  - Quando a Tag não existir ou não for do usuário: Disparo de erro de validação com mensagem exata (`"Tag não encontrada"`), caindo no mesmo handler.

## 5. Passos de Implementação Recomendados

1. **Atualizar `ITagRepository` e `TagRepository`** (se ausente):
   - Adicionar método para buscar múltiplas tags por IDs e Owner (ex: `findTagsByIdsAndOwner(tagsIds: string[], userId: string)`).
2. **Atualizar `ITaskRepository`**:
   - Refatorar a assinatura do método de busca, adicionando suporte ao DTO de filtros: `{ priority?: string; tags?: string[] }`.
3. **Atualizar `TaskRepository`**:
   - Adicionar a lógica de construção do objeto de consulta condicional (`priority` e `$in` em `tags`).
4. **Atualizar o Service**:
   - Adicionar a dependência do `ITagRepository` via construtor se necessária.
   - Implementar os *fail-fasts* lançando `400` antes de consultar os dados.
5. **Atualizar o `TaskController`**:
   - Realizar o cast do `req.query` de forma limpa.
6. **Configuração de Dependências** (Factory):
   - Atualizar a injeção na rota principal para garantir que os repositórios sejam passados corretamente na inicialização das classes.
