# RF12 — Plano Técnico: Agendamento de Job para Alerta de Tarefa

## Stack Tecnológica

- **Biblioteca de agendamento**: `node-schedule` — in-process, sem Redis ou banco externo.
  Permite agendar um job para uma data exata via `scheduleJob(date, callback)`.
- **Armazenamento de jobs**: `Map<string, Job>` em memória (`taskId → Job`).
  Jobs não persistem entre reinicializações do servidor — aceitável para o escopo do TCC.
- **Padrão de injeção**: `SchedulerService` entra via constructor no `TaskService`,
  seguindo o mesmo padrão de Clean Architecture já adotado no módulo.

## Modelo de Dados

Nenhuma alteração no schema Mongoose. O campo `alert?: string` já existe em
`src/infra/mongo/schemas/task.schema.ts` e armazena a data ISO 8601 como string.

## Arquivos novos

### `src/modules/task/port/scheduler-service.port.ts`
Contrato do agendador:
```typescript
export interface SchedulerServicePort {
  schedule(taskId: string, date: Date, callback: () => void): void;
  cancel(taskId: string): void;
}
```

### `src/infra/scheduler/scheduler.service.ts`
Implementação com `node-schedule`:
- `private jobs = new Map<string, nodeSchedule.Job>()`
- `schedule`: cancela job anterior se existir, agenda novo com `nodeSchedule.scheduleJob`
- `cancel`: cancela e remove da Map
- Exporta instância singleton: `export const schedulerService = new SchedulerService()`

### `src/modules/notification/port/notification-service.port.ts`
Port placeholder para RF13 — define o contrato que o job usará ao disparar:
```typescript
export interface NotificationServicePort {
  createFromAlert(taskId: string, ownerId: string): Promise<void>;
}
```

## Arquivos modificados

### `src/modules/task/services/task.service.ts`
- Adicionar `AlertInPastError` ("O horário do alerta deve ser uma data futura")
- Injetar `SchedulerServicePort` no constructor
- `createTask`: se `alert` definido → validar data futura → persiste task → agenda job
- `updateTask`:
  - Se `alert` no body → validar data futura
  - Após persistir: `scheduler.cancel(taskId)` + `scheduler.schedule(...)` se novo alert
  - Se `alert` removido/null: apenas cancela
- `deleteTask`: `scheduler.cancel(taskId)` antes de `taskRepository.delete`

**Callback do job**: recebe `taskId` e `ownerId` capturados em closure; em RF12 o
callback chama `notificationService.createFromAlert(taskId, ownerId)` (que será
implementado em RF13; na RF12 basta o port estar estruturado).

### `src/modules/task/infra/controllers/task.controller.ts`
- Importar `AlertInPastError`
- Handlers `create` e `update`: tratar `AlertInPastError` → `400`

### `src/modules/task/factories/task.factory.ts`
- Importar `schedulerService` (singleton) e `notificationService` (singleton placeholder)
- Injetar nos construtores de `TaskService`

## Tratamento de Erros

| Situação | Erro | HTTP | Mensagem |
|---|---|---|---|
| `alert` no passado | `AlertInPastError` | 400 | "O horário do alerta deve ser uma data futura" |
| Tag não encontrada | `TagNotFoundError` | 400 | "Tag não encontrada" |
| Tarefa não encontrada | `TaskNotFoundError` | 400 | "Tarefa não encontrada" |
| Erro interno | `Error` genérico | 500 | "erro interno do servidor" |
